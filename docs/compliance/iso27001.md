# ISO 27001:2022 — Annex A Technological Controls (A.8)

## Overview

ISO 27001:2022 is the international standard for Information Security Management Systems (ISMS). The 2022 revision restructured Annex A into four themes: Organisational (A.5), People (A.6), Physical (A.7), and Technological (A.8). Annex A.8 contains 34 controls directly addressing technical implementation.

This assessment covers **Annex A.8 only** — the subset addressable by infrastructure automation.

## Why It's Relevant

ISO 27001 is the most widely recognised information security standard globally. Enterprise procurement teams and regulated clients commonly require ISO 27001 certification or demonstrable alignment as a vendor pre-qualification criterion. For a hosting infrastructure, documented A.8 alignment provides:

- Evidence of systematic security configuration management
- Demonstrable vulnerability and patch management
- Comprehensive logging and audit capability with remote retention
- Formal access control and authentication standards

## Executive Summary

**Estimated Annex A.8 coverage: ~82% (known) ~ ~88% (assumed)**

The range reflects controls satisfied by external architecture that must be validated per deployment: MFA at VPN/bastion (A.8.5), block device encryption at the VM hypervisor layer (A.8.24), and web-layer controls at Cloudflare and ModSecurity (A.8.21). The lower figure reflects scripts alone; the upper figure reflects the full validated deployment.

Strong technical coverage across access management, authentication, vulnerability management, logging, network security, and configuration management. The multi-layer perimeter (Cloudflare → Edge FW → ModSecurity → Host) provides defence-in-depth across the network security controls. Administrative access via VPN/bastion with MFA satisfies authentication controls at the appropriate access boundary.

The cattle/pets architecture satisfies A.8.13 (backup) at the system design level — not a gap, a different DR strategy. SDLC controls (A.8.25, A.8.28) are partially met; this is an infrastructure tooling project, not an application under a formal secure development lifecycle.

| Control | Layer | What Provides It |

### Architectural Exceptions

| Control | Position | Compensating Controls |
|---|---|---|
| A.8.13 — Server-level backup | Servers are stateless cattle. No application data of record lives on the host OS. DR strategy is IaC re-deployment (~20 min) plus external application-layer data. Server backup is inappropriate for this architecture. | External replication; application backup; IaC rebuild; local disk volume risk tracked in gaps.md |
| A.8.7 — Host anti-malware (AV) | Host-level AV is inappropriate for a container-only host operating as a pod equivalent. AV cannot meaningfully scan container overlay filesystems and introduces false positives and operational risk. The equivalent controls exist at the correct layers: Trivy scans images before deployment, AppArmor + seccomp constrain container behaviour at runtime, and ModSecurity + Cloudflare filter web-layer threats. | Trivy image scanning; AppArmor docker-default; seccomp default profile; ModSecurity WAF; Cloudflare WAF; automatic OS updates; signed CI/CD artefacts |

### Optional Controls

Controls implemented in the default baseline but which may be disabled in specific deployment contexts:

| Control | Script | When May Be Skipped |
|---|---|---|
| A.8.5 — SSH MFA at host level (TOTP) | `setup-ssh-mfa.sh` | When the VPN/bastion enforces MFA and host-level TOTP would conflict with automated tooling. Not removing protection — MFA is at the access boundary. |
| GRUB bootloader password | `harden-bootloader.sh` | VM deployments where the hypervisor controls the boot sequence, or where automated reboots are managed externally. Reduces physical console risk only. |

## Why Excluded Today

Full ISO 27001 certification requires an accredited certification body to audit the complete ISMS — including organisational controls (A.5), people controls (A.6), physical controls (A.7), a Statement of Applicability, risk register, and management review cycle. This project delivers the technical layer (Annex A.8); the organisational governance programme is outside its scope and must be driven by the deploying organisation.

Within Annex A.8, the remaining partial coverage areas (A.8.25 SDLC, A.8.28 secure coding) reflect that this is a deployment automation toolkit rather than an application codebase. Formalising an SDLC for infrastructure scripts yields diminishing returns relative to the controls already in place (BATS tests, shellcheck CI, `sshd -t`/`visudo -c` validation in scripts).

Open technical gaps are tracked in [gaps.md](gaps.md).
