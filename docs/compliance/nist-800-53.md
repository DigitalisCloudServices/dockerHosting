# NIST SP 800-53 Rev 5 — Selected Control Families

## Overview

NIST SP 800-53 Rev 5 is the comprehensive US federal security control catalogue, providing controls across 20 families. It is the foundation for FedRAMP, StateRAMP, and FISMA compliance, and is widely referenced in civilian enterprise security programmes.

This assessment covers **eight selected families** directly addressable by infrastructure automation. The remaining twelve families (AT, CA, MA, MP, PE, PL, PM, PS, RA, SA, SR, and more) require organisational governance, procurement policies, or physical controls that are outside the scope of these scripts.

## Why It's Relevant

NIST 800-53 provides the most comprehensive publicly available control taxonomy for technical infrastructure. It is used as a reference framework for vendor security assessments, US government contracts (FedRAMP), and organisations seeking a structured approach to security control coverage regardless of direct regulatory obligation. Its control families for audit, access, and communications protection are strong reference points for any well-run infrastructure programme.

## Executive Summary

**Estimated coverage across assessed families: ~75% (known) ~ ~80% (assumed)**

The range reflects controls satisfied by external architecture pending deployment validation: MFA at VPN/bastion (IA-2), perimeter network controls at Cloudflare and ModSecurity (SC family), block device encryption at the VM hypervisor (SC-28), and platform-level container egress filtering (SC-7).

Strongest coverage in AU (Audit and Accountability) and SC (System and Communications Protection), driven by the comprehensive auditd ruleset, NewRelic remote log shipping, and the four-layer perimeter architecture. The MFA requirement (IA-2) is satisfied at the VPN/bastion access layer. CP (Contingency Planning) is satisfied architecturally — the cattle/pets model is the DR strategy, not a gap.

The main partial-coverage areas are IR (Incident Response — no formal runbooks, which is an organisational deliverable) and IA (no centralised identity provider, which is appropriate for the current scale).

## Control Family Assessment

### AC — Access Control (~78%)

| Control | Status | Implementation |
|---|---|---|
| AC-2 — Account Management | ✓ | Per-site user accounts; no shared accounts; `setup-docker-permissions.sh` |
| AC-3 — Access Enforcement | ✓ | Per-command sudoers; no docker group; file permissions 600/700 |
| AC-6 — Least Privilege | ✓ | No docker group; site users cannot access other sites |
| AC-7 — Unsuccessful Login Attempts | ✓ | PAM pam_faillock: 5 attempts → 15-min ban |
| AC-17 — Remote Access | ✓ | Key-only SSH; SSH behind VPN/bastion; no direct internet exposure |
| AC-18 — Wireless Access | N/A | Server deployment; no wireless interfaces |
| RBAC / ABAC beyond sudo | ~ | No centralised identity provider; per-site local accounts. Appropriate for current scale; would need revisiting for a larger fleet |

### AU — Audit and Accountability (~92%)

| Control | Status | Implementation |
|---|---|---|
| AU-2 — Event Logging | ✓ | auditd 28+ rules: auth, privilege escalation, filesystem, network, Docker — `setup-audit.sh` |
| AU-3 — Content of Audit Records | ✓ | auditd captures: timestamp, event type, subject, action, outcome, source IP |
| AU-4 — Audit Log Storage Capacity | ✓ | logrotate with defined retention; NewRelic remote storage |
| AU-9 — Protection of Audit Information | ✓ | Immutable auditd ruleset (`-e 2`); requires reboot to modify |
| AU-11 — Audit Record Retention | ✓ (conditional) | NewRelic provides remote retention; **NR account must be configured for ≥12 months** — see [gaps.md](gaps.md) |
| AU-12 — Audit Record Generation | ✓ | Comprehensive auditd ruleset + Docker json-file logging |
| Real-time log analysis / SIEM | ~ | NewRelic provides log ingestion and alerting; SIEM-style correlation requires alert policy configuration (operational, not infrastructure) |

### CM — Configuration Management (~82%)

| Control | Status | Implementation |
|---|---|---|
| CM-2 — Baseline Configuration | ✓ | Full IaC; every configuration is version-controlled code |
| CM-3 — Configuration Change Control | ~ | Git history provides audit trail; no formal CAB or change approval workflow |
| CM-6 — Configuration Settings | ✓ | All hardening via scripts; `sshd -t` and `visudo -c` validation |
| CM-7 — Least Functionality | ✓ | Docker-only model; no unnecessary services on host |
| CM-8 — System Component Inventory | ~ | Scripts define what is installed; no formal CMDB |

### IA — Identification and Authentication (~82%)

| Control | Status | Implementation |
|---|---|---|
| IA-2 — Identification and Authentication (MFA) | ✓ (at access layer) | MFA enforced at VPN/bastion before SSH is reachable; host TOTP available as defence-in-depth |
| IA-4 — Identifier Management | ✓ | Per-site dedicated users; per-command sudoers |
| IA-5 — Authenticator Management | ✓ | SSH key-only; PAM 14-char complexity + history; no password login over SSH |
| IA-6 — Authentication Feedback | ✓ | SSH login failures logged; fail2ban alerts |
| Centralised IdP | ~ | Not implemented; local accounts per host. Appropriate at current scale; fleet expansion toward K8s should adopt centralised identity (e.g., OIDC/LDAP). |

### SC — System and Communications Protection (~85%)

| Control | Status | Implementation |
|---|---|---|
| SC-5 — Denial of Service Protection | ✓ | Cloudflare L3/L4/L7 DDoS; SYN cookies; fail2ban |
| SC-7 — Boundary Protection | ✓ | Four-layer perimeter (Cloudflare → Edge FW → ModSecurity → UFW) |
| SC-8 — Transmission Confidentiality | ✓ | TLS 1.2/1.3 AEAD only at Cloudflare edge and Traefik; SSH with strong ciphers |
| SC-10 — Network Disconnect | ✓ | SSH `ClientAliveInterval`/`ClientAliveCountMax` in `harden-ssh.sh` |
| SC-12 — Cryptographic Key Management | ~ | TLS managed by Let's Encrypt / Cloudflare; no formal key management programme |
| SC-28 — Protection of Information at Rest | ✓ (at correct layers) | Block device encryption (VM hypervisor); application-layer encryption (applications); OS layer not the appropriate control point |
| SC-39 — Process Isolation | ✓ | Docker network isolation; per-site user isolation; userns-remap (optional) |
| Container egress filtering | ~ | Docker bypasses UFW at host level; platform-level and network-level external egress controls (Edge Firewall, VM network policy, data centre controls) are assumed to be in place — **validate per deployment**; host-level control tracked as [gaps.md](gaps.md) G1 |

### SI — System and Information Integrity (~75%)

| Control | Status | Implementation |
|---|---|---|
| SI-2 — Flaw Remediation | ✓ | `unattended-upgrades` daily; Trivy image scanning at deploy |
| SI-3 — Malware Protection | ✓ (at appropriate layers) | Architectural exception: host AV not appropriate; Trivy + AppArmor + seccomp + WAF — see [iso27001.md](iso27001.md) |
| SI-4 — System Monitoring | ✓ | NewRelic infrastructure agent; fail2ban; AIDE; auditd |
| SI-7 — Software, Firmware, and Information Integrity | ~ | AIDE daily batch FIM; no real-time integrity monitoring |

### CP — Contingency Planning (~72%)

| Control | Status | Implementation |
|---|---|---|
| CP-2 — Contingency Plan | ~ | DR approach is documented (IaC rebuild); no formal written contingency plan (organisational) |
| CP-9 — System Backup | ✓ (architectural) | Servers hold no application data of record; application backup + external replication is the data protection strategy; server backup is inappropriate for stateless hosts |
| CP-10 — System Recovery and Reconstitution | ✓ | `setup.sh` on a fresh Debian Trixie host restores full capability in ~20 minutes |
| Local volume data | ~ | Known transitional risk: Docker workloads may use local volumes; this data is outside the IaC recovery model — see [gaps.md](gaps.md) |

### IR — Incident Response (~40%)

|---|---|---|
| IR-4 — Incident Handling | ~ | Detection and alerting infrastructure in place (NewRelic, fail2ban, AIDE); no formal incident response runbooks |

| IR-6 — Incident Reporting | ~ | Internal alerting in place; formal reporting procedures are organisational |
| IR-9 — Information Spillage Response | ~ | No documented response procedure |

## Why Excluded Today

NIST 800-53 is a US federal framework. There is no regulatory obligation to comply for a UK-based hosting operation. It is tracked as a reference framework to ensure comprehensive control coverage.

The IR (Incident Response) partial coverage reflects an organisational gap, not an infrastructure gap. The detection and alerting tooling is in place; formal runbooks, escalation procedures, and reporting workflows are operational responsibilities that belong with the team running these servers, not the infrastructure automation project.

CP (Contingency Planning) is addressed architecturally. The cattle/pets model with IaC rebuild is a valid, faster DR strategy for a stateless infrastructure layer. Local volume data is a known transitional risk tracked in gaps.md, with the target state being external volume mounts as the architecture evolves toward Kubernetes.

Open technical gaps are tracked in [gaps.md](gaps.md).
